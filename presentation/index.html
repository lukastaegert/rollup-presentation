Why would they
want
this?

ES6 modules
from a user's perspective

  • export const x = 'foo' instead of
    exports.x = 'foo'
  • import {x} from 'y' instead of
    x = require('y').x
  • import and export only allowed at the top level
  • import ... from does not allow dynamic module names

ES6 modules
from a module bundler's perspective

  • Can identify unused modules without executing code
  • Can remove code for unused exports a.k.a.
    tree-shaking
But wouldn't this
break
my library for anyone who doesn't use ES6 imports?
…yes.

How to make ES6 exports work…

package.json: 
{
  // ...
  "main": "dist/index.js", // Common JS version
  "module": "dist/index.mjs", // ES6 version
  // ...
}
"module" is automatically used
by the major ES6 bundlers
